Project 1: Web Server

 

You will design and implement a simple HTTP/1.1 compliant web server.
Your web server should be fully compatible with standard web browsers.
Undergrad students may work in pairs. Grad students must work individually.

Project Requirements

Basic Functions

 

A functional webserver must perform the following general actions:

·     Receive a request from a socket

·     Parse the request to find out what the sender wants

·     Find the desired data

·     Respond to the sender with the appropriate information via a socket

 

Your webserver will only support the commands GET and HEAD.

·     GET implies the sender wants a file from the webserver, and therefore your webserver must return the response header and the desired file

·     HEAD implies the sender just wants the header for a file from the webserver, and therefore your webserver will only return the response header of that file

Any other commands received will be responded to as a "501: Not Implemented" error


The way an HTTP request is made is defined here: 5: Request

 

The way to make a response to a request is defined here: 6: Response

 

Specific Requirements

 

Your program shall run in the default Java package (no package line). The main method will be in the MyWebServer.java file.

 

Your program shall take two command line arguments. The first will indicate the port number to use, and the second will serve as the root path for all filenames. For example, given the command java MyWebServer.java 8817 ~/myweb, the URL "/" will be "~/myweb/".

 

Your server shall support absoluteURI requests (all this means is that if the request begins with http, you strip off everything before the third '/')


Your server should fetch index.html when the URL is a directory. Note that the URL the sender gives doesn't necessarily have a trailing slash indicating that its a directory. Also, it is OK to return a 404 if you get a request for a directory and index.html does not exist.

 

Your server must support the If-modified-since header, which is defined here: 14.25: If-Modified-Since

The java.net.URL class may be used to parse the URL, but you may not use it to fetch objects.


Your responses must include the additional headers "Date", "Server" (make up a name), "Last-Modified", and "Content-Length". These headers are defined in the following HTTP/1.1 sections:

·     3.3: Date and Time Format

·     3.8: Product Tokens

·     14.29: Last-Modified

·     8: Persistent Connections (connections are persistent by default. The easiest way to divide responses when using persistent connections is to use the Content-length header)

·     4.4: Content-Length

 

Your server shall report the appropriate status code in its response to a request. The codes will include:

·     200 OK : Returned when everything is normal

·     304 Not Modified : Returned when a GET has the condition If-Modified-Since and the file hasn't been modified since that time

·     400 Bad Request : Returned if the request is not in the proper syntax

·     404 Not Found : Returned when the desired file is not found

·     501 Not Implemented : Returned when it can't perform the desired action (when something other than GET or HEAD is requested)

 

When an error code is returned in the header of the response, attach some simple HTML after the header to inform the user that the specific error has occurred.

 

Optional: If you wish, you can make your server multithreaded, spawning an additional thread to handle each connection request. This is worth an extra 5%, but will not be used to exceed the project's max worth of 100%.

Things You Don't Need To Do

 

Unless you have a masochistic side, you shouldn't be aiming to achieve a fully-compliant HTTP/1.1 server. We have tried to specify what is necessary to implement a functional web server, and generally that set is restricted to what is required to support Chrome connections. If your web server supports HEAD and GET with persistent connections and If-modified-since from Chrome, you have probably implemented everything we expect. When in doubt, ask. In particular, encodings, languages, Accept:, and other constructs, while important, are unnecessary for this project. With all that said, masochists are welcome to participate in class.

 

You do not need to support hex-encoded strings in Request URIs

 

Your server is not required to implement the content-type header. Chrome will recognize file types based on their extensions.

 

Getting help

 

Both Kurose and Ross and Learning Java are good references for this project. You should find chapters 8 and 11 of Learning Java to be quite useful for learning about threads and socket programming. Both books and the slides also have examples of simple web servers you may wish to start from.

 

For questions regarding Java classes and methods, use this reference: Java API

 

You may find the following classes useful: java.util.StringTokenizer, java.io.BufferedReader, java.io.PrintWriter, java.io.InputStreamReader, java.util.TimeZone, java.util.Hashtable, java.text.SimpleDateFormat, java.io.FileReader, java.io.File.

 

You may find SimpleDateFormat HTTPDateFormat = new SimpleDateFormat("EEE MMM d hh:mm:ss zzz yyyy"); particularly useful.

 

Some hints on building your implementation. Modular design and testing is key to this project. After you complete your design and start implementation, you should first check that you can parse requests by printing what you parse to the terminal. (Debugging output is easy in this project, because the debugging output goes to your terminal, while the real output goes over the socket.) Try typing in requests using telnet, then direct your web browser at your server and have it send requests. Once you are parsing requests properly, then you can work on producing responses. Keep printing important information to the screen, perhaps including everything but the body of the message, so you can see what is happening between the client and server. It's very useful to be able to telnet to your server, send the exact request that a web browser sent, and look at the response.

 

 

When testing your server, either disable the cache on your browser (buried in preferences somewhere), or be prepared to do shift-reload a lot.

 

If your server dies in an ugly manner, it may leave the web server port in an unusable state for a few minutes. Just wait until it's free again, or pick another port.

 

It's easier to implement non-persistent connections before persistent connections. The change is actually quite minimal once you have non-persistent connections working (although a little thought in advance may help).

Code Sample and More Java Guidence

 

The java.io package contains two classes, InputStream and OutputStream, from which most of the other classes in the package derive. If you want to read content from a file, you can use FileInputStream; If you want to read data from client(or browser), you can use BufferedReader; The difference between BufferedReader and InputStream is the former will read text from a character-input stream, buffering characters so as to provide for the efficient reading of characters, arrays, and lines. If you want to write some content back to the browser, you can use DataOutputStream, which can write byte data to the receiver .

Despite Java's stream processing classes is very complex and nausea, it also provide more choices for us. Be careful when writting Input/Output data stream, and handling any possible exceptions.
